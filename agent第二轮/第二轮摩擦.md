## 9. route=update 的语义切换：覆盖 → append-only（2026-05-15 决议）

### 9.1 用户原话（§7 §36-37 行）

> 就是现在每步 ≤60 字、最多 3 步，我觉得就是这样对于 update 是不实用的，比如这次 update，我理想的状态是：比如原来的 mech / anchor / step 都保留，新的 update 就在下面，用一种展开 or whatever 的合适形式展现。你要是每次都刷新（覆盖），那我觉得不好。可以对应修改 schema：**一次写入还是原来的限制**，但是对于**一个 card 后续总的**应该对应调整。

### 9.2 客观评估

- **反馈成立**。当前 B v2 的 `output_kind=patch` 是"字段级 diff，merge 时覆盖原 crystallization"——这违背"原卡是历史资产"的产品直觉。
- **真实痛点不在单次长度**（每条 mechanism / anchor / step 单次写 ≤200/20/60×3 仍合理）——**在于卡的累积**：一张 IC 在多轮 update 后总共应该是「原层 + 多个新增层」，而不是「最近一次重写」。
- **方向选择**：append-only updates 数组 > 重写式 patch。原因：(a) 保留语言肌理与诊断时间线（每次的 patch_reasoning 都是 metadata 资产）；(b) UI 折叠展开天然适配；(c) 单次写入仍守原 schema，门槛不降；(d) 与"双卡（update + meta）"摩擦（§8）解耦——这是同一份 raw 多次诊断的累积模型，自然支持。

### 9.3 决议方向（→ composer 实施，详见 `plan-update-append-mode.md`）

- **schema**：加可选 `updates: []` 数组（item 含 `updated_at` / `patch_reasoning` / 可选 `crystallization{m,a,steps}` / `patterns_added` / `source_refs_added` / `questions_appended` / `trigger_addendum`）；单次每字段仍守原限制
- **B prompt（v2.1，本次已改）**：`output_kind: patch → update_entry`；语义从 diff 改成 append；新写一条独立的 mechanism / anchor / micro_steps 层，**不复制原卡内容再改**
- **A prompt（v2.3，本次已改）**：`update_directives.*` 描述"新增哪一层"，不再写"保留/不动/可去"覆盖语义
- **merge（py，待 composer）**：`run_pipeline.py merge --mode update`——读 existing card，把 update_entry append 到 `card.updates[]`，`patterns_added`/`source_refs_added` 求并集，`questions_appended` / `trigger_addendum` 拼到原数组/字符串末尾；**原 crystallization 字段一字不动**
- **export（py，待 composer）**：`export_v3_chains.py` 序列化 `updates` 数组到 v3 md（折叠展开形式）、chains.json、chains.data.js
- **UI（待 composer）**：prototype 渲染原卡正面 + 折叠"更新历史 (N)"块；每条 update entry 列 updated_at + patch_reasoning + 新增的 mechanism/anchor/micro_steps（如有）

### 9.4 与 §7 / §8 的关系

- §7（"收摊/手柄"等怪喻 + micro_steps 写细）：在 append-only 下天然缓解——update 卡可在新 update_entry.micro_steps 里写新动作，不挤压原 3 步；同时单次仍守 ≤60×3，**写作纪律不滑坡**
- §8（同 md 双卡 update + meta）：append-only 不解决双卡——仍要两轮 A 调用（先 route=update 写 IC-012 的 update_entry，再 route=meta 出新元锚卡）；但两个 update_entry 可以来自不同 raw 时间点，自然累积
