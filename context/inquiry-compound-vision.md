# Inquiry Compound — 项目愿景与核心想法

> 这份文档是给 subagent / 后续执行 agent 的 onboarding 文件。它**只讲 what & why，不讲 how**（技术栈、UI、文件结构都在 `.cursor/plans/inquiry_compound_v0_*.plan.md` 里）。读完这一份你应该能理解：用户在解决一个什么真实痛点、产品的灵魂是什么、什么该做什么不该做。

---

## 1. 用户是谁、卡在哪

- 大三学生，平时遇到内耗、纠结、人生意义类困惑时，会和 Gemini / ChatGPT 进行**深度对话**（不是查资料那种，是 self-inquiry 那种）
- 这些对话在当下都给了他真实的启发，让他活得更松弛、更活在当下
- **问题在于 compound 失效**：
  - 几个月前想通的道理，过段时间又忘
  - 同一类情境再次发生，他又内耗，找不回当时的对话
  - 知道得越来越多，但内化得不够，反复"被打回原形"
- 他的直觉：如果能 systematically solidify all of these，他能成为一个明显更好的人

## 2. 用户已经在手动做的工作流（这是最重要的线索）

他已经无意识地发明了一套"问题复利"的手动流程：

- 2 月有一段有价值的对话 → 导出成 md
- 3 月遇到新事，**把 2 月的 md 粘进新对话**作为上下文
- 4 月又遇到事，**把 2 月 + 3 月都粘进去**
- 5 月直接问："根据这些，你看我的 underlying pattern 核心问题在哪、有没有我没意识到的事"
- **关键细节：他只导出自己问的问题，不导出 AI 的答案**——他朴素地相信"问题才是最有价值的，也节省 AI 的 attention"

这套流程跑通了，但很重，很慢，靠记忆翻聊天记录。这个工具就是**把这套流程从手动变成 30 秒动作**。

## 3. 产品的灵魂（一句话锁死）

> **"问题"是资产，"答案"是耗材。**
>
> 这个工具是一个 question library，按【底层心态 pattern】（而非表面话题）索引用户过去所有深度问题，两个出口共享同一个分类引擎：
>
> 1. **被生活 trigger 时** → 快速检索过去同 pattern 的问题和当时的顿悟，避免重新内耗
> 2. **要去 Gemini 聊新问题前** → 一键打包相关历史问题为 md，复制贴进 Gemini，复利之前的深度

这两个出口不是两个功能，是同一个引擎的两种用法。它们之间还有一个**反馈环**：旧的解不够 → 拎着旧的去问新的 → 新的对话又被沉淀回库里 → 库越来越深。**这才是 compound 真正发生的地方。**

## 4. 核心数据原子：Inquiry Chain（追问链）

不是扁平的"问题卡"。一个 md 是一条 chain：

- **Trigger**：当时是什么事让他开始问的（surface 情境）
- **追问路径**：他一步步追问的问题序列（只存他的问题，不存 AI 答案）
- **Crystallization**：他从这次对话里提炼出的核心顿悟 / 心得 / mental model
- **Pattern tags**：这条 chain 底层的心态 pattern（不是话题，是心理结构——比如"对他人评价的恐惧"、"试图获取通关攻略以逃避不确定"）

理解这个数据结构是理解整个产品的钥匙。**任何把它退化为"flat 问题列表"或"普通笔记"的设计都偏离了灵魂。**

## 5. 真正不可替代的核心：分类引擎

- 导入：现有 Chrome 插件已经做了，**我们不做**
- 存储：本地数据库，**不是技术难点**
- UI：搜索框 + 卡片 + 一键打包，**一周能写完**
- **分类（按底层心理 pattern）：唯一没人替我们做、也唯一决定工具好不好用的事**

如果分类做得好——用户输入"我又因为别人看法内耗了"——工具能拎出 5 个月前"被室友 judge"、3 个月前"导师评价"、2 周前"实习对比同学"，**因为它们底层是同一个 pattern**。这一刻用户会"啊原来我一直在解同一道题"——这就是他要的"underlying pattern 自我都没意识到的事"。

如果分类只按表面话题（室友/导师/实习），这就是个普通搜索工具，没有任何复利价值。

**整个产品的成败赌在这一件事上。**

## 6. 道德经原则（防漂移护栏）

来自用户认同的一篇文章 `外部source/道德经ai`，已经被锁定为产品和工程纪律：

- **不尚贤**：拒绝"AI 高手"身份焦虑。每天的成功标准是"build 了一个具体的小东西 + 自己用了一次"，不是"我学了多少新概念"
- **不贵难得之货**：拒绝囤工具。用最朴素的栈把流程跑顺。看到 LangChain / Mem0 / Pinecone / LangGraph / 任何"内部玩法" → 记下来跳过
- **不见可欲**：拒绝追 hype。不刷新模型新闻。每周钻 1 个底层概念（embedding / structured output / RAG / agent loop），其他全是噪音

工程上的体现：每一个新功能都要过三道闸：
1. 它解决的是哪条 `friction.md` 里的具体卡壳？解决不了 → 不做
2. 不加这个能不能用？能 → 暂缓
3. 它会不会引入新依赖 / 新概念 / 新"难得之货"？会 → 极高门槛

## 7. 用户的 build 哲学（必须遵守）

- **AI 写代码，用户钻原理**：用户没写过前端，AI 负责实现细节；用户的时间花在底层概念（embedding 是什么、structured output 是什么、context engineering 怎么做）
- **Build > theory**：动手永远比读综述重要。"准备好了再开始"是 wishful thinking，是不行动的借口
- **一手资料**：尽量读官方文档 / 一手论文，不要博主二手解读
- **每周钻深 1 个**：不要广撒网。深度 > 广度
- **Dogfood 强制**：写完一段就要自己用，不允许"build 但不用"

## 8. 明确不做的事（v0 anti-feature 清单）

这些是用户最初想到但被一起 reduce 掉的功能。它们**不是"以后做"，是"必须等真实使用产生需求才允许做"**：

- ❌ Dashboard / 心态曲线 / 连续天数 / 成长 metrics
- ❌ AI 每日推送 / 每日 reminder / 每日生成内容
- ❌ 工具内对话（用户和 Gemini 的关系不要被打扰）
- ❌ 后端 / 登录 / 用户系统 / 分享 / 社交
- ❌ Career signaling / 简历用途 / 公开发布
- ❌ 成就系统 / 游戏化 / 评分

这些被砍掉**不是因为不重要，是因为它们是结果不是起点**。在 dogfood 自然产生具体痛点之前，做了就是装饰。

## 9. v0 之后的演进方向（仅作参考，不在当前范围）

- **v1**：涌现式 taxonomy。chains 累积到一定量，跑聚类 + LLM 给 cluster 命名，得到用户专属的"心理 pattern 宇宙"
- **v2**：Agent workflow。用户输入"被 X 困扰"，agent 自动决定检索深度、是否再追问、打包多大 context

v0 的架构应该为这两步留好钩子，但**不允许提前实现**。

## 10. 副作用：这是用户的 AI 工程训练营

用户是大三、想练 AI engineering 技能。这个项目恰好是**伪装成个人工具的真实 AI 工程训练**：

- v0 = structured output + embedding + 简单检索（核心 RAG 概念）
- v1 = clustering + emergent summarization（高阶 RAG）
- v2 = agent workflow + context curation（agentic 系统）

这给项目带来双重意义：**做出来用户自己受益（解决内耗）+ 简历上是一个连贯的 portfolio piece**。subagent 在执行时如果有教学时机，可以适度解释底层原理，但**不要让"教学"挤占"build"的时间**——build 本身才是最好的学习。
