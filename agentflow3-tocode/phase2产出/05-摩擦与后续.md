# 摩擦与后续

## 1. 与 plan §5.7 示例输出的差异

- Plan 示例含 `[route_helper] ok`、`[a] ok (route=new, ...)` 等**行级进度**；当前实现以 **阶段结束后** 的 `_print_summary` 汇总为主。若需要与 chat 日志逐行对齐，可在后续加 `print_fn` 钩子或 stderr 进度条（不改 merge 语义）。

## 2. `output_kind` 缺失

- Plan 文案倾向「缺失 → schema_fail / awaiting_human」。实现将 **`None` 视为 `full_card`** 以便兼容旧 B JSON；若 `output_kind` 为**非空非法字符串**，仍走 `awaiting_human` + `schema_fail`。

## 3. `Ctrl+C` 与 resume

- 中断时 `status=failed`，**不会**自动把「进行中的一步」标为完成；resume 从未完成 stage 重跑。与 plan「不重跑已完成」一致；若需在 SIGINT 时区分 `interrupted` 状态，可后续加枚举值（Phase 4 inbox 再定）。

## 4. `--debug-dir`

- Phase 1 CLI 支持用户指定 `--debug-dir`；orchestrator 固定为 `runs/<id>/_debug/`。若需透传，可在 `run_single_case` 增加可选参数（小改动）。

## 5. `manifest.human_override` 与 Phase 4

- `awaiting_human` 时 `human_override` 仍为 null；inbox **accept** 后 Phase 4 可写 `human_override` 并 `--resume --force-pass`（本实现已支持 `force_pass` 写 `judge.accepted.json`）。

## 6. 后续 Phase 衔接

- **Phase 3**：直接 `from agents_runtime.orchestrate import run_single_case`（`no_push=True` 批跑）。
- **Phase 4**：扫描 `runs/*/manifest.json` 的 `status` / `next_action` / `verdict`；字段名保持本 phase 约定。
